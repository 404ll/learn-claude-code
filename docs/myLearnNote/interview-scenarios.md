# Agent 系统设计场景题

从仓库学习概念 → 扩展为专业叙述的思路梳理。

---

## 场景一：大型项目代码审查 Agent

### 题目

设计一个代码审查 agent，需要：
1. 读取大型项目里的所有文件
2. 对每个文件写出审查意见
3. 最后汇总成一份报告

跑了一会儿发现 agent 开始"犯糊涂"，前面读过的文件内容开始影响后面的判断，而且越跑越慢。如何解决？

### 思路拆解（从仓库概念出发）

- **根本原因**：单 agent 的 context 窗口无法承载大型项目的全量信息，随着读取文件增多，context 膨胀导致性能下降和推理质量劣化
- **每个文件审查是独立任务** → 用 subagent（s04）隔离执行，用完即销毁，主 agent 只收摘要
- **三个阶段有依赖关系** → 用任务图（s07）的 `blockedBy` 管理：单文件审查可并行，汇总任务等所有审查完成才解锁
- **主 agent context 还是会涨** → 加 context 压缩（s06）兜底

### 专业表述

这个问题的根本原因是**单 agent 的 context 窗口无法承载大型项目的全量信息**，随着读取文件增多，context 膨胀导致性能下降和推理质量劣化。

解决思路是**任务拆解 + 并行隔离执行**：

1. **将每个文件的审查拆分为独立子任务**，用独立的 agent 实例执行。每个子任务上下文干净、互不干扰，执行完只返回摘要结果给主 agent——这是经典的 **Map-Reduce 模式**：先分散执行（Map），再汇总（Reduce）。

2. **子任务之间的依赖关系用 DAG（有向无环图）管理**：单文件审查任务之间互相独立可并行，汇总报告任务依赖所有审查任务完成后才能启动。

3. **主 agent 的 context 通过摘要压缩保持可控**：子任务只向主 agent 返回结构化摘要，不把完整的执行过程暴露给主 agent。

**核心原则：隔离执行，只在边界处交换结果**——子任务的噪音不污染主流程，主流程只关心最终结论。

## 场景二：长期运行的编程助手 Agent

### 题目

设计一个长期运行的编程助手 agent，用户可以随时发新任务。有时候任务很简单（改个变量名），有时候很复杂（重构一个模块）。复杂任务可能要改很多文件，简单任务希望秒回。如何设计？

### 思路拆解（从仓库概念出发）

- **任务复杂度差异大** → 需要两条执行路径，不是所有任务都走同一套机制
- **简单任务秒回** → 直接在主 agent 执行，不派发（派发本身有开销，反而更慢）
- **复杂任务改很多文件** → 用 worktree 隔离（s12），独立分支操作，出问题可丢弃
- **复杂任务耗时长** → 后台异步执行（s08），主 agent 不阻塞，用户可继续发任务
- **长期运行、随时发任务** → 持久化任务队列（s07），存到磁盘，重启不丢失，空闲时自动认领（s11）

### 专业表述

核心挑战是**任务复杂度差异大 + 需要长期运行**，设计上需要区分两条执行路径。

**判断路径：**
主 agent 作为调度层，根据任务复杂度决定执行策略——简单任务直接执行，避免引入不必要的调度开销；复杂任务走隔离执行路径。

**复杂任务的三个保障：**
1. **目录隔离**：用 Git worktree 给每个复杂任务分配独立工作区和分支，并行执行互不干扰，出问题可以直接丢弃分支
2. **非阻塞执行**：复杂任务异步后台运行，主 agent 不阻塞，用户可以继续发新任务
3. **持久化任务队列**：任务存到磁盘，agent 空闲时自动认领，重启或 context 压缩后任务不丢失

**核心设计原则：按需分配复杂度**——不是所有任务都需要复杂机制，引入隔离、异步、持久化本身有成本，只在真正需要的时候用。架构的价值在于按需分配复杂度，而不是把所有任务都走最重的路径。

## 从仓库概念到专业表述的转换规律

|----------|----------|

| 任务图 blockedBy（s07） | DAG（有向无环图）依赖管理 |
